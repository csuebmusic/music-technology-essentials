# Modules

Week-by-week (or session-by-session) lesson plans for MUS 381.

Each module file is the **canonical lesson plan** — the "spec" for what that session is. It's the document anyone can read in 5 minutes to understand what students do, what concepts are introduced, and what they leave with.

Operational detail — block-by-block facilitation, demo scripts, common confusions, pacing fallbacks, what to assess — lives in the parallel file in [`ta-notes/`](../../ta-notes/). Module plans link out to TA notes; they don't duplicate that content.

## File naming

`module-XX-week-YY.md` — e.g. `module-01-week-01.md`, `module-02-week-02.md`. Modules can span multiple weeks, in which case there's one file per week within the module.

## Standard sections

Every module file should include:

- **Module purpose** — one paragraph on why this module exists
- **Learning outcomes** — numbered list of what students should be able to do by the end
- **Session structure** — table of blocks with timing and one-line focus each
- **Key concepts introduced** — short list of the conceptual takeaways
- **Deliverable** — what students produce, where it goes
- **Listening assignment** — what they listen to, when it's due (or a note that there isn't one this week)
- **Student-facing materials** — links to relevant handouts
- **What to bring to next session** — short, practical

What does NOT go here:

- Step-by-step demo scripts (TA notes)
- Common student confusions (TA notes)
- Pre-class prep checklists (TA notes)
- Pacing fallbacks (TA notes)
- Detailed order-of-operations procedures (TA notes)
- Q&A for likely student questions (TA notes)

If a future TA wants to teach the module, they read the TA notes. If Inés (or any reviewer, accreditor, or future course coordinator) wants to know what the course actually does, they read the module plan.
